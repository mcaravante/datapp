# `@cdp/db`

Prisma schema, migrations, and client for the CDP.

## Bootstrap (first-time)

```bash
# 1. Make sure Postgres is running
docker compose -f infra/compose/docker-compose.dev.yml up -d postgres

# 2. From the repo root
cp .env.example .env   # fill in secrets / connection strings
pnpm install

# 3. Create the initial migration (do not apply yet)
pnpm prisma -F @cdp/db migrate dev --create-only --name init

# 4. Append the `real_revenue` generated-column conversion to the init
#    migration's migration.sql (see "real_revenue" section below).

# 5. Apply it
pnpm db:migrate

# 6. Seed regions + default tenant
pnpm db:seed
```

Subsequent schema changes:

```bash
# Edit prisma/schema.prisma
pnpm prisma -F @cdp/db migrate dev --name <descriptive_name>
```

## `real_revenue` is a Postgres generated column

The `order.real_revenue` column is materialized in the database as:

```sql
GENERATED ALWAYS AS (total_invoiced - total_refunded) STORED
```

Prisma cannot express this purely in `schema.prisma`. The discipline is:

1. The schema declares `realRevenue Decimal? @db.Decimal(20, 4)` — a regular
   nullable column.
2. The initial migration converts it to a `GENERATED ALWAYS AS ... STORED`
   column. Append this to `prisma/migrations/<timestamp>_init/migration.sql`:

   ```sql
   ALTER TABLE "order" DROP COLUMN "real_revenue";
   ALTER TABLE "order"
     ADD COLUMN "real_revenue" NUMERIC(20, 4)
     GENERATED ALWAYS AS ("total_invoiced" - "total_refunded") STORED;
   ```

3. **Never INSERT or UPDATE `realRevenue` from the application.** Prisma will
   error (`cannot insert into column ... it is a generated column`).
4. **Never run `prisma db pull`** — introspection will rewrite the schema and
   could surface the GENERATED expression in a non-portable way.

Read it back as a normal column: `prisma.order.findMany({ select: { realRevenue: true } })` works.

## UUIDv7

All primary keys use UUIDv7 via Prisma's `@default(uuid(7))` (Prisma 5.14+
generates the value client-side). UUIDv7 is monotonically increasing, which
keeps B-tree indexes friendly even as we scale.

## Multi-tenancy

Every domain table carries `tenantId` and a composite index that starts with
it. **Never write a query that omits `tenantId` from a WHERE clause** (a
NestJS interceptor in `apps/api` enforces this at runtime — see ADR 0004).

## Layout

```
packages/db/
├── prisma/
│   ├── schema.prisma            # Single source of truth for the data model
│   ├── seed.ts                  # 24 INDEC provinces + default tenant
│   └── migrations/              # Generated by `prisma migrate dev`
├── generated/client/            # Prisma client output (gitignored, regenerated by `pnpm db:generate`)
└── package.json
```

The package's main + types fields point straight at `generated/client/`,
so consumers `import { PrismaClient } from '@cdp/db'` and receive the
generated CommonJS module — no wrapper layer. Lifecycle (connect /
disconnect) is owned by the consumer; in `apps/api` a `PrismaService`
extends the client and hooks Nest's `OnModuleInit` / `OnModuleDestroy`.

## Postgres extensions

The schema requires these extensions (declared via `datasource db.extensions`):

| Extension  | Purpose                                               |
| ---------- | ----------------------------------------------------- |
| `pgcrypto` | `gen_random_uuid()`, envelope encryption helpers      |
| `pg_trgm`  | Trigram fuzzy matching for customer name/email search |
| `citext`   | Reserved for future case-insensitive columns          |
| `unaccent` | Argentine names with diacritics in search             |

The first migration `CREATE EXTENSION IF NOT EXISTS` for each.
